# ADR-0001: Runtime State via `data-bob-*` Attributes Instead of BEM Modifiers

**Status**: Accepted  
**Date**: 2026-03-03  
**Deciders**: BlazOrbit

---

## Context

BlazOrbit components originally expressed runtime state (disabled, loading, error, active, selected, visible, etc.) using BEM-style CSS class modifiers on the component root, e.g.:

```html
<bob-component class="bob-button bob-button--disabled bob-button--loading">
```

This pattern created several problems:

1. **Dual source of truth**: the component C# code toggled classes, the CSS selected on those classes, and tests asserted on both. Changing a class name required touching three places.
2. **Builder coupling**: `BOBComponentAttributesBuilder` emits state attributes automatically from `IHas*` interfaces. There was no equivalent automatic emission of CSS classes — each component manually concatenated strings in its `.razor` or `.cs` file.
3. **Selector specificity wars**: scoped `.razor.css` selectors like `.bob-button--disabled` competed with global family CSS and consumer overrides. Blazor isolation added `[b-xxx]` scope attributes, but modifier classes still had unpredictable specificity when mixed with attribute selectors.
4. **BEM pollution**: as the component library grew, the number of `--*` modifiers exploded. Many were only used in a single CSS rule.

## Decision

**Runtime state on the `<bob-component>` root shall be expressed exclusively via `data-bob-*` HTML attributes.** BEM modifiers (`--selected`, `--focused`, `--active`, etc.) are permitted on **child elements inside a component** but forbidden on the root.

Examples of the new pattern:

```html
<!-- Before -->
<bob-component class="bob-button bob-button--disabled bob-button--loading">

<!-- After -->
<bob-component data-bob-component="button"
               data-bob-disabled="true"
               data-bob-loading="true">
```

CSS selectors target the attribute:

```css
[data-bob-component="button"][data-bob-disabled="true"] button {
    opacity: var(--bob-opacity-disabled);
}
```

## Consequences

### Positive

- **Single source of truth**: `BOBComponentAttributesBuilder` owns the emission. Components declare `IHas*` interfaces; no manual string concatenation.
- **Uniform specificity**: attribute selectors have flat, predictable specificity. No more specificity wars between scoped `.razor.css`, global family CSS, and consumer overrides.
- **Smaller DOM**: boolean attributes are **omitted when `false`** (see ADR-0005). The root element carries only the state that is currently active.
- **Test uniformity**: tests assert against `data-bob-*` attributes consistently, regardless of which component is under test.
- **Lintable**: a single regex (`CssArchitectureLintTests.RazorCss_Should_Not_Use_Bem_State_Modifiers_On_Root`) guards the rule.

### Negative

- **Child elements still need BEM**: state on children (dropdown items, tab triggers, picker cells, tree nodes) does not go through `BOBComponentAttributesBuilder`. BEM modifiers remain the pragmatic choice there.
- **Attribute selectors are slightly more verbose** than class selectors in hand-written CSS. This is offset by the fact that most state CSS is generated by `BlazOrbit.BuildTools`.
